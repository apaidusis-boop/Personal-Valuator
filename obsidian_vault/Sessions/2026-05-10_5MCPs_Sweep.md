## 1. Markdownify — wired in 3 places (the biggest change)

### What changed
Created `library/_md_extract.py` as a single helper that knows how to extract text from PDF / DOCX / XLSX / PPTX / HTML / images / audio via Microsoft `markitdown`, with `pdfplumber` and `pypdf` as safety-net fallbacks.

Three call sites now accept an `engine` arg / CLI flag:

| Site | Before | After |
|---|---|---|
| `monitors/cvm_pdf_extractor.py::extract_text` | hard-coded pdfplumber | `--engine {pdfplumber, markitdown, auto}` (default pdfplumber for backwards compat) |
| `library/ingest.py::_extract_pdf` | hard-coded pypdf | `--engine {pypdf, markitdown, auto}` (default pypdf for backwards compat) |
| `fetchers/subscriptions/_pdf_extract.py::extract_pdf_text` | hard-coded pypdf | `engine=` kwarg (default pypdf for backwards compat) |

**Defaults are unchanged** — surgical principle (no implicit migration). Opt-in via flag or `II_PDF_ENGINE=markitdown` env var.

### Empirical bench A — 5 CVM PDFs (clean text-based)

```
            chars   speed
pdfplumber  9473    0.41s   (baseline for CVM extractor)
pypdf       9631    -       (baseline for library + subscriptions)
markitdown  9825    0.71s   (+3.7% / +2.0% chars, ~2x slower but still <1s)
```

Markitdown wins on **structure** (Markdown headers, tables, lists) more than raw char count. Side-by-side samples saved in `reports/markitdown_samples/1_*.{txt,md}`.

### Empirical bench B — Subscription PDFs (`reports/markitdown_subs_bench.json`)

| PDF | size | pdfplumber | pypdf | markitdown |
|---|---|---|---|---|
| xp_top_dividendos | 608 KB | 18 678 chars | 18 776 | **19 422 (+4%)** |
| suno_carteiras | 346 KB | **17 chars** | 35 | 19 |
| suno_dividendos_ed23 | 1.9 MB | **0 chars** | 8 | 8 |
| suno_tese_itsa4 | 11.8 MB | **0 chars** | 56 | 56 |

**🔥 Critical finding (not the markitdown story):** Suno's PDFs are
**image-based** (rasterized) — pdfplumber, pypdf, AND markitdown all
return ~0 useful text. The XP PDF is text-based and works fine.

This means the existing `ii subs extract` pipeline has been silently
returning empty insights on Suno PDFs. Markitdown doesn't fix this on
its own — it would need OCR. Two follow-up paths:

1. **OCR layer** — add Tesseract or Azure Document Intelligence as a
   markitdown plugin (`markitdown[az-doc-intel]`) or pre-process via
   `pytesseract`. Cost: ~5-15s per page. Gates: needs Azure key OR
   tesseract install.
2. **Scrape Suno HTML reports instead of PDFs** — `PlaywrightSession`
   already authenticated for Suno via cookies; switch the SunoAdapter
   to fetch the HTML rendering instead of the PDF download. No OCR
   needed; better tables.

Path 2 is the cleaner fix. Logged for next session.

## 2. Playwright — public-portal scraper (NEW)

### What was already there
`fetchers/subscriptions/_session.py::PlaywrightSession` already exists for cookie-auth sites (Suno, XP, WSJ, Finclass, Fool). Battle-tested, persistent profiles in `.playwright-profiles/`.

### What's new
`fetchers/portal_playwright.py` — a **public-only** scraper for non-authed sites. Combines Playwright JS rendering with markitdown HTML→MD conversion.

### Capabilities
- Fetch JS-heavy pages (SPA frontends) that `requests` can't see
- Wait for specific selectors (SPA hydration)
- Output rendered HTML, optional Markdown via markitdown, optional full-page PNG screenshot
- 24h cache by default in `data/portal_cache/<sha1>.{html,md,png,meta.json}`
- Single CLI for one-off due-diligence on any URL

### How to use
```bash
# Render fiis.com.br property page + convert to MD
python fetchers/portal_playwright.py "https://www.fiis.com.br/knhf11/" --md

# RI page with screenshot
python fetchers/portal_playwright.py "https://ri.itausa.com.br" --md --screenshot

# Wait for selector (SPA)
python fetchers/portal_playwright.py "https://app.b3.com.br/..." --md --selector "table.results"

# Fresh fetch (bypass cache)
python fetchers/portal_playwright.py "<url>" --md --ttl 0
```

### Use cases unlocked
- B3 calendar / company filing index that requires JS
- fiis.com.br lazy-loaded property data and rent rolls
- Status Invest / Fundamentus pages where raw HTML is incomplete
- One-off due-diligence on RI sites the user mentions in chat
- Screenshotting the Mission Control views or Streamlit dashboard for thumbnails

### What's still required for portals like XP / JPM / Suno
The new scraper does **not** attempt logins (no creds in this session, and that requires user-controlled cookies). The existing `PlaywrightSession` flow stays the channel for authed scraping — Cookie-Editor export → `data/subscriptions/cookies/<source>.json` → `ii subs fetch`.

3. **Did not test Context7 or Task Master end-to-end.** Both are stdio MCPs that Claude Code only loads on startup; testing requires a restart. Smoke-test commands listed above.

4. **Did not attempt logins to XP / JPM / Suno via Playwright.** No creds in this session and that's a separate authed flow. Existing `PlaywrightSession` pipeline (cookie export) is the canonical path.
